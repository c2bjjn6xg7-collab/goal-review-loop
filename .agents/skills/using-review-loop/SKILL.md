# Using Review-Loop

How to operate the `review-loop` CLI to run multi-agent AI development tasks.

**Core workflow:** Init → Start → Monitor → Retry/Resume → Verify → Merge.

## First-Time Setup in a New Project

If `review-loop` is not installed globally yet:

```bash
cd /path/to/review-loop-source    # the cc劳工系统 repo
npm install -g .                   # install review-loop CLI globally (creates symlink)
```

Verify it works:
```bash
review-loop --version
```

In the target project (must be a git repo with clean worktree):

```bash
cd /path/to/your-project
review-loop init                   # generates review-loop.yaml + .gitignore (.agent/** ignored)
```

Default config uses opencode planner + claude developer + codex auditor + parallel enabled.

### Prerequisites

- **Git repo**: target project must be a git repository with clean working tree
- **AI CLI installed**: at least one of `claude`, `codex`, or `~/.opencode/bin/opencode`
- **review-loop CLI**: installed globally (symlink — auto-updates on `npm run build`)

### review-loop.yaml

Generated by `review-loop init`. Defines 4 agent roles and their commands:

```yaml
agents:
  planner:        # 规划师，拆任务（默认 opencode/deepseek）
  developer:      # 开发者，写代码（默认 claude/glm-5.2）
  auditor:        # 审计员（默认 codex）
  final_auditor:  # 最终复核（默认 codex）

parallel:
  enabled: true
  max_parallel_workers: 2

loop:
  max_iterations: 3
  max_agent_retries: 3
```

## Updating review-loop

Since global install is a symlink, just rebuild the source:

```bash
cd /path/to/review-loop-source    # cc劳工系统 repo
git pull
npm install
npm run build                     # global review-loop auto-updates (symlink)
```

No need to re-run `npm install -g .` unless the symlink is broken.

## Quick Start

```bash
review-loop start --request "..." --parallel   # Start a task
review-loop dashboard                          # Browser: live output + events + cancel
```

## Commands

| Command | Purpose |
|---|---|
| `review-loop start --request "..." --task-slug "..."` | Start a development task |
| `review-loop start --watch` | Start with inline terminal progress |
| `review-loop status --watch` | Terminal event stream monitor |
| `review-loop dashboard` | Browser dashboard (SSE live output + timeline + cancel) |
| `review-loop resume` | Resume an interrupted run (Ctrl+C / crash) |
| `review-loop retry` | Retry a BLOCKED run from failure point (skips passed tasks) |
| `review-loop retry --force` | Retry without confirmation prompt |
| `review-loop cancel` | Cancel active run |
| `review-loop clean` | Remove stale state.json/lock/worktrees |
| `review-loop clean --dry-run` | Preview what would be cleaned |
| `review-loop config agents` | Interactive: view/switch role→model |
| `review-loop config agents --set planner=claude` | Non-interactive model switch |
| `review-loop providers list` | List registered agent providers |

## Switching Models

Each role (planner/developer/auditor/final_auditor) can use a different AI CLI:

```bash
review-loop config agents --set planner=opencode/ownplan/deepseekv4pro
review-loop config agents --set developer=claude
review-loop config agents --set auditor=codex
```

## CRITICAL: Known Pitfalls

### 1. opencode binary path

**Never use `npx opencode-ai`** — the npm package ships only a Windows launcher, not a macOS binary.

**Always use `~/.opencode/bin/opencode`** — the native binary installed by opencode's own installer.

Verify before use:
```bash
file ~/.opencode/bin/opencode  # Should show: Mach-O 64-bit executable arm64
```

### 2. yargs flag parsing

CLI agents that use yargs (opencode, possibly others) will parse `--foo` patterns inside prompt text as CLI flags. This causes immediate help output + exit 1.

**Always add `--` before the prompt argument in agent commands:**
```
# WRONG:
opencode run --model X --no-replay "$P"
# RIGHT:
opencode run --model X --no-replay -- "$P"
```

This is enforced in `src/cli/config.ts` for the opencode provider.

### 3. Git worktree must be clean

`review-loop start` requires a clean git worktree. If a previous run left uncommitted changes:

```bash
git checkout -- .
git clean -fd
review-loop clean
review-loop start --request "..."
```

### 4. Stale run state blocks new runs

If `state.json` has a non-terminal phase from a crashed/interrupted run, new runs will auto-recover stale locks (no manual clean needed). But if state itself is corrupted:

```bash
review-loop clean  # Removes state.json, run.lock, cancel-request.json, worktrees/
```

### 5. events.jsonl is per-run isolated

Each run gets a fresh `events.jsonl`. Previous runs are auto-archived to `.agent/history/events-<runId>.jsonl`. Do not manually delete `events.jsonl` — the orchestrator handles archival.

### 6. review-loop.yaml format

The file has a `.yaml` extension. The `config agents` command handles parsing/writing correctly. If editing manually, preserve the existing format.

### 7. Don't touch the worktree while a run is in progress

If a review-loop run is active, do NOT run `git checkout -- .`, `git clean -fd`, or `rm -rf .agent/worktrees/`. These will corrupt the active worktree and cause developer agents to fail. If a run is stuck, use `review-loop cancel` or `review-loop retry` instead.

### 8. Idle watchdog now monitors stdout only

The idle watchdog (aborts stalled developers after 480s) only monitors stdout and handoff file growth, NOT stderr. This is intentional — heartbeat messages on stderr would keep the watchdog alive forever. If a developer produces no stdout for 8 minutes, it will be aborted and retried.

## Recovery: When a Run Fails

### Run BLOCKED (task failed, audit rejected)

```bash
review-loop retry
```

This reads the BLOCKED state, shows which tasks passed (skip) and failed (retry), resets failed tasks to pending, and resumes. No manual cherry-pick needed.

If retry fails too:
```bash
review-loop retry --force   # skip confirmation
# or start fresh:
review-loop clean
review-loop start --request "..." --parallel
```

### Run interrupted (Ctrl+C / crash / power off)

```bash
review-loop resume    # picks up from where it left off
```

Stale locks are auto-recovered — no need to `review-loop clean` first.

### Integration audit REWORK (auditor found issues)

The system **automatically** enters a rework loop: developer gets auditor feedback, re-runs, re-audits, up to `max_iterations` times. No manual intervention needed.

## Monitoring a Run

### Browser Dashboard (recommended)

```bash
review-loop dashboard --port 8800
# Open http://127.0.0.1:8800
```

Shows: current phase, agent status, stage progress bar, event timeline, live agent output (filtered), artifacts, run selector, cancel button.

### Terminal

```bash
review-loop status --watch          # Human-readable
review-loop status --watch --json   # JSON Lines (for external tools)
```

### What you'll see during a run

Events fire at **phase boundaries**, not continuously:
- `run.started` → `role.started planner` → [planner runs 2-5 min, heartbeat every 30s] → `role.exited` → `task.started` → [developer runs 5-10 min] → `task.completed` → `role.started auditor` → ... → `run.completed`

Quiet periods between events are normal — the agent is working. Heartbeat events (every 30s) confirm the agent process is alive.

## Parallel Execution

Parallel mode is enabled by default (`parallel.enabled: true` in review-loop.yaml).

Planner splits tasks into parallelizable groups. Independent tasks run concurrently in isolated git worktrees. Tasks with dependencies run sequentially after their prerequisites complete.

```bash
# Override worker count:
review-loop start --request "..." --parallel --max-parallel-workers 4
```

## After a Run Completes

```bash
# Check the result
git log --oneline -3                          # Review-loop commits to a feature branch

# Independently verify (don't trust PASS blindly):
git checkout agent/<run-id>-<task-slug>
npm run typecheck && npm run lint && npm test -- --run

# Merge to main:
git checkout main
git merge --no-ff agent/<branch> -m "..."
git tag -a <phase>-reviewed -m "..."
git branch -d agent/<branch>
```
