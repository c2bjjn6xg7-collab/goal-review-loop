# 需求文档：Codex + Claude Code Goal Review Loop 改造方案

> 文档定位：本文件定义产品目标、业务流程、角色边界、功能范围和验收标准。
> 具体模块、状态机、文件格式和实现约束见《DECT落地设计文档.md》。

## 1. 项目背景

当前开发流程如下：

1. 使用 Codex / GPT 高能力模型完成需求理解、项目规划、任务拆解和技术方案设计。
2. 使用 Claude Code，并通过 cc switch 选择低成本模型或国产模型完成具体编码。
3. 开发完成后，使用 Codex 审查真实代码改动、测试结果和交接说明。
4. 审计不通过时，将明确的问题清单交还 Claude Code 定向返工。
5. 返工完成后由 Codex 再次审查，直至通过或达到最大返工次数。
6. 审计通过后才允许创建本地 Git commit，并可创建本地 tag。

当前痛点：

* 需要手动切换模型、复制需求、传递审计意见。
* 开发、审计、返工之间缺少统一的文件协议。
* 模型可能自行宣布完成，但实际 diff、测试或需求并未通过。
* 缺少稳定的执行状态和跨会话恢复能力。
* 缺少受控的本地 Git 分支、提交和版本追踪流程。

本项目需要基于现有项目 `claude-code-review-loop` 进行改造，形成可自动执行、可中断恢复、可审计的本地开发闭环。

## 2. 建设目标

实现一套以任务为单位的 Goal Review Loop：

```text
用户输入需求
  ↓
Codex 规划并生成 .agent/plan.md、.agent/GOAL.md
  ↓
系统记录 Git 基线和执行状态
  ↓
Claude Code /goal 执行开发
  ↓
执行 GOAL.md 中规定的验证命令
  ↓
生成 .agent/developer-handoff.md
  ↓
Codex 审查 GOAL、handoff、验证结果和真实 git diff
  ↓
生成 .agent/audit-report.md
  ↓
FAIL：Claude Code 仅按审计问题返工，最多 3 轮
  ↓
PASS：Codex 生成 .agent/final-audit.md
  ↓
系统创建本地 Git commit 和可选 tag
```

最终系统应做到：

* 一条命令启动完整流程。
* 每个阶段都有明确输入、输出和责任主体。
* 开发模型不能自行决定最终完成状态。
* 审计必须以真实代码、真实 diff 和真实命令退出码为依据。
* 流程中断后可以从最近的合法阶段继续。
* 只有最终审计通过的版本才能提交到本地 Git。

### 2.1 交付形态：CLI 核心 + Codex 插件入口

本项目的核心交付物是本地 CLI 引擎 `review-loop`，而不是把全部能力直接写进 Codex
插件。CLI 负责状态机、模型调用、范围校验、验证执行、Git 证据采集和最终提交；Codex
插件或 Skill 只作为用户入口，负责把自然语言任务转交给 CLI。

目标形态：

```text
用户
  ├─ 终端直接执行 review-loop
  └─ Codex Desktop / Codex CLI / IDE Extension 中调用 Review Loop 插件
        ↓
      插件 Skill 读取任务并调用本机 review-loop CLI
        ↓
      CLI 编排 Planner / Developer / Auditor
        ↓
      输出 .agent/ 证据、状态和最终结论
```

这样保留原 `claude-code-review-loop` 的优点：用户可以在 Codex 中用一句自然语言触发
“Claude 执行、Codex 审查”的工作流；同时避免把状态恢复、Git、验证和审计逻辑塞进一个
难以测试的 Skill 文件中。

使用方式分三层：

1. 开发者或高级用户直接使用 CLI：`review-loop init`、`review-loop start`、
   `review-loop status`、`review-loop resume`、`review-loop cancel`。
2. Codex 用户通过插件/Skill 使用：在 Codex Desktop 或 Codex CLI 中说“用 Review Loop
   实现这个需求”，插件调用本地 CLI。
3. 后续可选包装：提供 marketplace/plugin 元数据，让用户安装后无需记忆底层命令。

## 3. 核心设计原则

### 3.1 规划、开发、评估职责分离

Codex 负责高价值判断：

* 理解需求和识别风险。
* 生成总体计划和可执行 Goal。
* 定义成功标准、允许范围、禁止范围和验证命令。
* 审查真实 diff、验证结果和开发交接说明。
* 输出返工要求和最终审计结论。

Developer CLI（Claude Code、CodeBuddy、OpenCode、国产模型 CLI 等）负责受控执行：

* 按 GOAL.md 修改代码。
* 运行规定的验证命令。
* 如实记录实现、测试和阻塞情况。
* 审计失败后只修复报告中列出的问题。

编排器负责流程控制：

* 管理状态、轮次、命令调用和超时。
* 采集 diff、文件清单、验证结果和命令退出码。
* 校验文档格式和修改范围。
* 仅在最终通过后执行 commit/tag。

### 3.2 GOAL.md 是当前任务的执行契约

GOAL.md 必须至少包含：

* 任务目标。
* Success Criteria。
* Allowed Changes。
* Disallowed Changes。
* Verification Commands。
* 非目标和已知约束。

开发模型不得修改 GOAL.md。执行过程中如需改变 Goal，必须停止当前流程，由 Codex 重新规划并明确记录变更。

### 3.3 审计必须基于事实证据

Codex 审计必须读取：

* `.agent/plan.md`
* `.agent/GOAL.md`
* `.agent/developer-handoff.md`
* `.agent/verification/` 下的命令、输出和退出码
* 从任务基线到当前工作区的完整 Git diff
* 未跟踪文件清单及其内容或摘要
* 修改范围校验结果

不能仅根据开发模型的自然语言说明判定通过。

### 3.4 状态由系统维护

`.agent/state.json` 是当前任务的机器状态，由编排器维护。

开发模型和审计模型不得直接修改：

* `.agent/state.json`
* `.agent/GOAL.md`
* `.agent/audit-report.md`
* `.agent/final-audit.md`

模型可以生成的文件由阶段决定：

* Codex 规划阶段：`plan.md`、`GOAL.md`
* Claude 开发阶段：`developer-handoff.md`
* Codex 审计阶段：`audit-report.md`、`final-audit.md`

### 3.5 一次只处理一个 Goal

单次运行只允许存在一个活动 Goal。不得在同一轮中顺带实现未列入 Success Criteria 的功能。

### 3.6 验证通过不等于最终通过

自动化验证证明命令执行成功；Codex 审计还必须判断：

* 是否满足全部 Success Criteria。
* 是否存在逻辑错误、回归或安全问题。
* 是否越过 Allowed Changes。
* 是否修改了禁止文件。
* 是否删除、跳过或弱化测试。
* handoff 是否与真实 diff 一致。

### 3.7 默认保护用户工作区

MVP 默认要求：

* 当前目录必须是有效 Git 仓库。
* 必须存在可用的 HEAD 基线提交。
* 启动任务前工作区必须干净。

如果不满足，流程应停止并给出原因，不得自动执行 `git restore`、`git clean`、`git reset --hard` 等破坏性命令。

## 4. 范围定义

### 4.1 MVP 必须实现

1. 初始化和环境预检。
2. Codex 规划调用与产物校验。
3. Claude Code `/goal` 开发调用。
4. GOAL 验证命令执行和结果持久化。
5. developer-handoff 生成与格式校验。
6. 基于任务基线的完整 diff 采集。
7. Allowed/Disallowed Changes 范围校验。
8. Codex 审计调用和 PASS/FAIL/BLOCKED 解析。
9. 最多 3 轮返工。
10. 状态持久化与 `resume` 恢复。
11. 最终审计报告。
12. 审计通过后的本地 commit 和可选 tag。
13. 全流程 iteration log。

### 4.2 暂不纳入 MVP

* 自动 push、自动创建远程 PR。
* 多个 Goal 并行执行。该能力后续作为 Phase 7 的多 worktree 调度层实现。
* 多仓库联动修改。
* 自动合并分支。
* Web 管理后台。
* Codex 插件市场包装和 GUI 外壳。
* Agent 自行修改流程规则或自动进化 Prompt。
* 在脏工作区中自动区分用户改动和 Agent 改动。
* 无人工确认地执行破坏性 Git 回退。

## 5. 角色与权限

| 角色 | 可以做 | 不可以做 |
|---|---|---|
| 用户 | 提供需求、启动/停止/恢复流程、修改配置 | 无 |
| 编排器 | 调用模型、运行验证、写状态、采集 diff、最终 commit/tag | 自行修改业务代码、绕过失败验证 |
| Codex Planner | 生成 plan.md、GOAL.md | 修改业务代码、提交代码 |
| Developer CLI | 修改 Allowed Changes、运行验证、写 handoff | 修改 Goal/状态/审计结论、扩大范围、提交代码 |
| Codex Auditor | 读取事实证据、写 audit/final audit | 修改业务代码、直接提交代码 |

最终 PASS 必须由 Codex Auditor 给出，并通过编排器的机械校验后才生效。

### 5.1 Claude Code 权限策略

Claude Code 的授权分为两类：

1. **账号/模型授权**：由用户在本机 Claude CLI 中完成，例如运行 `claude auth` 或按 Claude
   CLI 的提示完成登录。`review-loop` 不保存 Claude 账号凭据，也不在 `.agent/` 中记录密钥。
2. **工具执行权限**：由 `review-loop.yaml` 中 Developer 的命令模板决定，例如使用
   `--permission-mode acceptEdits`、`--permission-mode bypassPermissions` 或
   `--dangerously-skip-permissions`。

默认策略必须保守：

* 默认使用 `acceptEdits` 或等价的受控权限模式。
* 只有用户明确配置并确认目标仓库可信、可回滚、无敏感生产数据时，才允许开启 bypass。
* bypass 模式不得由 Planner、Developer 或 Auditor 自行切换；只能由用户配置或初始化向导写入。
* 即使 Claude 处于 bypass 模式，Scope Guard、Verification Runner、Artifact Schema、Git
  preflight 和最终审计仍必须照常执行。

推荐配置层级：

| 场景 | Developer 命令策略 |
|---|---|
| 默认安全模式 | `claude -p --permission-mode acceptEdits`，通过 stdin 输入 prompt |
| 可信本地沙盒 | `claude -p --permission-mode bypassPermissions`，通过 stdin 输入 prompt |
| 完全无人值守实验 | `claude -p --dangerously-skip-permissions`，仅限隔离仓库和用户显式选择 |
| 精细授权模式 | 使用 `--allowedTools` / `--disallowedTools` 限定工具范围 |

文档、CLI 输出和插件入口都必须明确提示：bypass 不是最终安全边界，只是减少 Claude
交互确认；真正的安全边界仍是本系统的范围校验、验证、审计和 Git 基线保护。

### 5.2 Developer Provider 扩展策略

本系统不得与 Claude Code 强绑定。Claude Code 是默认 Developer Provider，但后续必须支持
CodeBuddy、OpenCode、GLM、Qwen、Gemini CLI 或其他本地编程 CLI。

当前阶段可通过 `review-loop.yaml` 的 `agents.developer.command` 接入任意 CLI，但要满足：

* 支持非交互输入任务，最好支持从 stdin 或 prompt file 读取。
* 能在指定项目目录中运行。
* 能真实修改文件，而不是只输出建议。
* 能返回稳定退出码。
* 能按提示生成 `.agent/developer-handoff.md`。

后续应增加正式 Provider Profile：

| 字段 | 说明 |
|---|---|
| `provider_id` | `claude`、`codebuddy`、`opencode`、`custom` 等 |
| `display_name` | 展示给用户的名称 |
| `command_template` | 实际执行命令 |
| `prompt_transport` | `stdin`、`prompt_file`、`argv` |
| `health_check` | 如 `claude --version`、`codebuddy --version`、`opencode --version` |
| `permission_modes` | 支持的权限模式或空列表 |
| `transcript_mode` | stdout/stderr、JSONL、无结构文本等 |
| `supports_non_interactive` | 是否适合无人值守调用 |

建议新增 CLI：

```bash
review-loop providers list
review-loop providers test claude
review-loop providers test codebuddy
review-loop providers test opencode
review-loop config set developer.provider opencode
```

插件入口不得直接决定调用哪个编程 CLI；它应把用户选择写入配置或传给 `review-loop`，由
CLI 统一执行、记录和审计。

### 5.3 智能模型路由与并发调度愿景

后续系统应支持在项目开始时对任务进行复杂度和并发性分类，把不同任务交给不同能力和成本的
Developer Provider：

* 架构设计、跨模块重构、安全敏感、数据迁移、复杂调试等高风险任务，交给更强模型或更可靠的
  Provider。
* 单文件小改动、补测试、文档、简单样式、机械性重命名等低风险任务，可交给低成本国产模型或
  custom Provider。
* 需求不清、验收标准不足或影响范围未知的任务，先交给强 Planner 做澄清和拆解，再决定开发模型。

推荐抽象为 worker，而不是写死某个 Plus 插件：

| worker | 职责 |
|---|---|
| `planner` | 总规划、任务拆解、依赖和复杂度评估 |
| `premium_worker` | 高复杂度、高风险、高返工成本任务 |
| `balanced_worker` | 普通功能和中等复杂度修复 |
| `cheap_worker` | 低风险、小范围、重复性任务 |
| `auditor` | 子任务审计和返工判断 |
| `integration_auditor` | 多任务合并后的最终审计 |

该能力不应让模型自由选择自己要做什么，而应由系统生成结构化任务图：

```text
Project Request
  → Task Graph / DAG
  → Complexity Classifier
  → Provider Router
  → Worktree Scheduler
  → Per-task Review Loop
  → Integration Audit
```

并发规则：

* 同一个工作区仍只能有一个活动 run，继续由 `.agent/run.lock` 防止并发写入。
* 可并发任务必须在独立 Git worktree 和独立任务分支中执行。
* 只有依赖关系为空、Allowed Changes 不重叠、验证命令可独立运行的任务，才能并发。
* 所有并发任务通过后，必须由 Integration 阶段顺序合并，并重新运行全量验证和审计。
* 出现 merge conflict、共享文件冲突、全量验证失败或集成审计失败时，不得自动强推或静默覆盖。

升级规则：

* Low/Medium 问题优先退回原 worker 返工。
* High/Critical 问题升级给 `premium_worker`。
* 同一 cheap 任务连续失败 2 次，升级给 `premium_worker`。
* 需求拆解或架构方向错误时，退回 `planner` 重新规划。

该能力属于 Phase 7 或更后续阶段，不进入 Phase 1-6 的本地单任务闭环。

## 6. 文件与目录结构

```text
project-root/
├─ review-loop.yaml               # 项目级配置，可选；缺省时使用安全默认值
└─ .agent/
   ├─ plan.md                     # 当前任务总体计划
   ├─ GOAL.md                     # 当前任务执行契约
   ├─ developer-handoff.md        # 最新一轮开发交接
   ├─ audit-report.md             # 最新一轮审计报告
   ├─ final-audit.md              # 最终审计报告
   ├─ iteration-log.md            # 追加式流程日志
   ├─ progress.json               # 桌面端/插件读取的机器进度
   ├─ progress.md                 # 用户可读进度摘要
   ├─ state.json                  # 编排器状态
   ├─ run.lock                    # 防止同一项目并发运行
   ├─ transcripts/                # Agent 输出摘录和可读转写
   │  ├─ iteration-01-developer.md
   │  └─ iteration-01-auditor.md
   ├─ verification/
   │  ├─ manifest.json            # 验证结果汇总
   │  └─ iteration-01/
   │     ├─ command-01.stdout.log
   │     └─ command-01.stderr.log
   └─ history/
      └─ iteration-01/
         ├─ developer-handoff.md
         └─ audit-report.md
```

固定文件表示“当前最新版本”；进入下一轮前，将上一轮 handoff 和 audit 归档到 `history/`，避免覆盖历史证据。

### 6.1 版本化策略

最终通过后，默认随业务代码进入 Git commit 的文件：

* `.agent/plan.md`
* `.agent/GOAL.md`
* `.agent/developer-handoff.md`
* `.agent/audit-report.md`
* `.agent/final-audit.md`

仅用于本地运行、默认写入 `.gitignore` 的文件：

* `.agent/state.json`
* `.agent/run.lock`
* `.agent/iteration-log.md`
* `.agent/verification/`
* `.agent/evidence/`
* `.agent/history/`
* `.agent/debug/`

这样既保留最终开发契约和审计结论，又避免运行日志、锁文件和终态状态在 commit 后继续造成脏工作区。若项目需要长期归档完整证据，应由后续配置增加独立归档目录或制品存储，不应直接提交可能包含敏感信息的大体积日志。

## 7. 文件职责

### 7.1 plan.md

由 Codex Planner 生成，描述：

* 需求理解。
* 当前项目现状。
* 技术方案。
* 模块和步骤拆分。
* 风险及验证策略。

### 7.2 GOAL.md

由 Codex Planner 生成，是开发和审计共同遵守的契约。关键字段必须使用固定格式，供编排器解析。

### 7.3 developer-handoff.md

由 Claude Developer 生成，至少记录：

* 本轮状态：COMPLETED 或 BLOCKED。
* 实际修改文件。
* 实现说明。
* 执行过的验证。
* 已知风险。
* 未解决问题。

开发模型在 handoff 中声明 COMPLETED 只代表“提交审计”，不代表最终通过。

### 7.4 audit-report.md

由 Codex Auditor 每轮生成，必须包含机器可解析结论：

* `PASS`
* `FAIL`
* `BLOCKED`

FAIL 时每个问题必须包含严重级别、证据、影响和可执行的返工要求。

### 7.5 final-audit.md

由 Codex Auditor 在最终通过或流程终止时生成。结论固定为 `PASS`、`FAILED` 或 `BLOCKED`，并记录：

* 最终结论。
* 完成的 Success Criteria。
* 最终验证结果。
* 修改范围。
* 已接受的剩余风险。
* 本次运行标识和 diff 摘要。

### 7.6 iteration-log.md

由编排器追加写入，记录每个阶段的开始时间、结束时间、状态、轮次、失败原因和恢复行为。模型不得重写历史记录。

### 7.7 state.json

由编排器原子写入，至少包含：

* `run_id`
* `phase`
* `iteration`
* `max_iterations`
* `base_commit`
* `branch`
* `last_error`
* 各阶段状态和时间

## 8. 标准执行流程

### 8.1 初始化和预检

编排器应检查：

1. Git 仓库、HEAD、当前分支和工作区状态。
2. Codex、Claude Code 和配置命令是否可用。
3. `.agent/run.lock` 是否表示已有活动进程。
4. 配置文件和超时参数是否合法。
5. `.agent` 是否存在未结束任务。

预检失败时不得启动模型开发。

### 8.2 规划阶段

1. 收集用户需求和必要的项目上下文。
2. 调用 Codex Planner。
3. 生成 `plan.md` 和 `GOAL.md`。
4. 校验 GOAL 固定字段、路径规则和验证命令。
5. 记录 `base_commit`。
6. 创建或切换本地任务分支。

### 8.3 开发阶段

Claude Code 的目标提示必须要求：

1. 读取 plan.md 和 GOAL.md。
2. 只实现当前 Goal。
3. 只修改 Allowed Changes。
4. 不修改 Disallowed Changes 和控制文件。
5. 不提交代码。
6. 完成后生成 developer-handoff.md。
7. 无法完成时明确返回 BLOCKED。

### 8.4 验证阶段

验证命令由编排器执行，不以模型声称的结果为准。

每条命令必须记录：

* 原始命令。
* 工作目录。
* 开始和结束时间。
* 退出码。
* 标准输出和错误输出文件。
* 是否超时。

任一必需命令失败时，本轮不得进入 PASS。

### 8.5 审计阶段

编排器先完成机械检查：

* 必需文件是否存在且格式合法。
* 是否有禁止文件被修改。
* 是否有超出 Allowed Changes 的文件。
* 必需验证是否全部成功。
* 是否存在删除或弱化测试的可疑改动。

然后调用 Codex Auditor。Auditor 必须逐项检查 Success Criteria，并输出 audit-report.md。

### 8.6 返工阶段

audit 结果为 FAIL 时：

1. 归档当前 handoff 和 audit。
2. 轮次加一。
3. 将 GOAL.md、最新 audit-report.md 和真实 diff 提供给 Claude Code。
4. 只允许修复报告列出的问题及其必要测试。
5. 重新执行完整 Verification。
6. 重新审计。

达到最大轮次仍未通过时，状态改为 FAILED，并生成 final-audit.md，不执行 commit。

### 8.7 最终通过和提交

同时满足以下条件才允许 commit：

* 所有必需验证命令成功。
* 范围校验成功。
* 最新 audit-report 的结论为 PASS。
* final-audit.md 已生成且结论为 PASS。
* 工作区仍与被审计 diff 一致。

提交规则：

* 只创建本地 commit。
* commit 包含业务改动及第 6.1 节列出的版本化审计文档。
* 本地运行文件必须已被正确忽略且不得曾被 Git 跟踪。
* commit message 包含 run_id 或任务标识。
* 可配置创建本地 tag。
* 默认不 push。

## 9. Git 管理规则

### 9.1 基线

每个任务开始时记录当前 `HEAD` 为 `base_commit`。后续审计均比较 `base_commit` 与当前工作区，而不是只比较最近一次模型调用。

### 9.2 分支

一个任务使用一个本地任务分支，所有返工轮次在同一分支完成。建议命名：

```text
agent/<run-id>-<task-slug>
```

不得为每次返工创建新分支。

### 9.3 diff 范围

审计证据必须覆盖：

* 已跟踪文件的修改和删除。
* 新增但未跟踪的文件。
* 文件重命名。
* 二进制文件清单。

### 9.4 禁止行为

系统不得默认执行：

```bash
git reset --hard
git clean -fd
git restore .
git push
git push --force
```

如需清理失败任务，必须提供独立、明确、需要用户确认的操作。

---

## 10. 状态与恢复

流程阶段至少包括：

```text
INITIALIZING
PLANNING
DEVELOPING
VERIFYING
AUDITING
REWORKING
FINALIZING
PASSED
FAILED
BLOCKED
CANCELLED
```

每次阶段变化前后都必须更新 state.json 和 iteration-log.md。

`resume` 应：

1. 读取 state.json。
2. 校验 Git 分支、base_commit 和工作区是否仍匹配。
3. 检查上一个阶段的产物是否完整。
4. 从最近一个安全阶段继续。
5. 不得重复 commit 或跳过验证。

---

## 11. 异常处理

以下情况必须进入 BLOCKED 或 FAILED，不得伪装成 PASS：

* 模型命令不存在、认证失败、超时或异常退出。
* 模型未生成要求的文件。
* 文档缺少机器可解析结论。
* 验证命令失败或超时。
* 修改超出允许范围。
* Git 基线变化、分支被切换或工作区被外部修改。
* 达到最大返工次数。
* 审计结论与机械校验冲突。

机械校验失败时，即使 Codex 输出 PASS，最终结果也必须是 FAIL 或 BLOCKED。

---

## 12. 配置要求

模型调用命令、超时、最大轮次和 Git 策略必须可配置，不能硬编码到业务流程。

配置至少支持：

* Planner 命令模板。
* Developer 命令模板。
* Auditor 命令模板。
* 每类调用的超时时间。
* 最大返工次数，默认 3。
* 分支和 tag 命名模板。
* 是否自动创建 commit，默认是。
* 是否创建 tag，默认可配置。
* 工作区必须干净，MVP 固定为是。

cc switch 负责 Claude Code 所使用的具体模型，Review Loop 只负责调用配置后的 Claude 命令，不与某个国产模型强绑定。

后续智能调度配置应支持：

* Provider 能力档位、成本等级和并发上限。
* 任务复杂度阈值与模型路由规则。
* 每个 Provider 的速率限制、预算限制和失败降级策略。
* Git worktree 根目录和最大并发任务数。
* 禁止并发的路径模式，例如数据库 migration、核心配置、安全策略文件。

这些配置只在 Phase 7 调度层启用，不能影响 Phase 1-6 的单 Goal 安全闭环。

## 13. 验收标准

### 13.1 正常通过

* 能从用户需求生成合法 plan.md 和 GOAL.md。
* 能调用 Claude Code 完成开发并生成 handoff。
* 能真实执行 Verification 并保存退出码和日志。
* 能审查从 base_commit 到当前工作区的完整 diff。
* Codex PASS 后生成 final-audit.md。
* 最终只产生一个通过审计的本地 commit。
* 可选 tag 正确指向该 commit。
* 最终工作区干净。

### 13.2 审计失败和返工

* FAIL 后不会 commit。
* audit-report 中的问题能自动传递给下一轮开发。
* 每轮 handoff、audit 和验证日志均被保留。
* 最多执行配置的返工次数。
* 后续轮次仍执行完整验证和完整 diff 审计。

### 13.3 范围保护

* 修改禁止文件时机械校验失败。
* 修改 Allowed Changes 之外的文件时机械校验失败。
* 删除测试或跳过必需验证时不能通过。
* Developer 修改控制文件时不能通过。

### 13.4 中断恢复

* 在开发、验证、审计任一阶段中断后，可以通过 resume 恢复。
* 恢复前会检查 Git 和状态一致性。
* 不会重复创建分支、重复提交或丢失上一轮证据。

### 13.5 安全

* 脏工作区默认拒绝启动。
* 不自动执行破坏性 Git 命令。
* 不自动 push。
* 命令超时后能终止子进程并记录原因。

6. 实现开发、验证、审计主循环。
7. 实现返工、历史归档和最大轮次限制。
8. 实现 final audit、commit 和 tag。
9. 实现 resume、异常恢复和端到端测试。
10. 实现 Codex 插件入口、Provider Profile、进度和 transcript。

## 15. 完成定义

只有同时满足以下条件，本项目改造才算完成：
